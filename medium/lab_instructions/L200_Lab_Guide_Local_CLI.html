What You're Building

In this lab, you'll develop and test an AI agent locally on your machine, then deploy it to Databricks as a production application. The agent can search documents, query data, and remember conversations.

Document Search (RAG)

Semantic search over policy documents — finding answers by meaning, not just keywords.

Data Queries (Natural Language)

Ask questions about data in plain English. The agent translates to SQL and returns insights.

Conversation Memory

Persistent memory powered by Lakebase (PostgreSQL) — the agent remembers across sessions.

Local Dev → Cloud Deploy

Full dev loop: edit code locally, test instantly, then deploy to Databricks with one command.

Step 1

Clone the Repository

git clone <repo-url> cd <repo-name>
Step 2

Prepare Workshop Data

Before building the agent, you need to create the data resources it will use as tools.

Option A — Run the setup notebook in Databricks (recommended):

  1. Import the repo into your workspace (Repos → Add → Git Folder)
  2. Open data/workspace_setup_script/01_quickstart_setup.py
  3. Fill in catalog and schema widgets → Run All

Option B — Run local scripts via CLI:

cd data/local_cli_setup_script # Create data tables python execute_sql.py --profile DEFAULT --warehouse-id <WAREHOUSE_ID> # Chunk policy documents python execute_chunking.py --profile DEFAULT --warehouse-id <WAREHOUSE_ID>
Option B Limitation
With Option B, you still need to manually create the Vector Search endpoint+index, Genie Space, and MLflow experiment. See data/README.md for details.
Important — Save These Values
Note the outputs from the setup: MLflow Experiment ID, Vector Search Index name (e.g., catalog.schema.policy_docs_index), Genie Space ID. You'll need them later.
Step 3

Authenticate with Databricks CLI

# Log in to your workspace databricks auth login --host https://<your-workspace>.cloud.databricks.com # Verify it worked databricks auth profiles

This creates a profile (default: DEFAULT) that all subsequent commands will use.

Step 4

Run Quickstart

This one command sets up your local environment:

cd medium uv run quickstart --profile DEFAULT

It creates a .env file, sets up an MLflow experiment, and updates databricks.yml.

Already have an experiment?
Pass it directly: uv run quickstart --profile DEFAULT --experiment-id <ID>
Step 5

Create a Lakebase Instance

Lakebase provides the persistent memory for your agent. Create an autoscaling project in the Databricks UI or via CLI.

  1. Go to Compute → Lakebase → Create Project (Autoscaling)
  2. Name it (e.g., my-agent-workshop)
  3. Wait for it to provision (~1-2 minutes)
  4. Run Cell 3 of medium/scripts/lakebase_setup_script.ipynb to find the database ID

Note down the branch path and database path from the output.

Step 6

Configure Environment

Edit your .env file to add Lakebase configuration:

# .env file in the medium/ directory DATABRICKS_CONFIG_PROFILE=DEFAULT MLFLOW_EXPERIMENT_ID=<your-experiment-id> CHAT_APP_PORT=3000 CHAT_PROXY_TIMEOUT_SECONDS=300 MLFLOW_TRACKING_URI="databricks" MLFLOW_REGISTRY_URI="databricks-uc" # Lakebase (agent memory) LAKEBASE_AUTOSCALING_ENDPOINT=projects/<project>/branches/<branch>/endpoints/<endpoint> LAKEBASE_AGENT_MEMORY_SCHEMA=agent_openai_memory # Frontend Lakebase connection (chat history) PGHOST=<your-lakebase-hostname> PGDATABASE=databricks_postgres PGPORT=5432 PGUSER=<your-email@company.com>

Then update databricks.yml with your resources:

# databricks.yml — resources section resources: apps: agent_openai_agents_sdk: name: "agent-<your-app-name>" # Choose a unique app name description: "OpenAI Agents SDK agent application" source_code_path: ./ config: command: ["uv", "run", "start-app"] env: - name: MLFLOW_TRACKING_URI value: "databricks" - name: MLFLOW_REGISTRY_URI value: "databricks-uc" - name: API_PROXY value: "http://localhost:8000/invocations" - name: CHAT_APP_PORT value: "3000" - name: CHAT_PROXY_TIMEOUT_SECONDS value: "300" - name: MLFLOW_EXPERIMENT_ID value_from: "experiment" # ← pulled from resource below - name: LAKEBASE_AUTOSCALING_ENDPOINT value_from: "postgres" # ← pulled from resource below - name: LAKEBASE_AGENT_MEMORY_SCHEMA value: "agent_openai_memory" - name: PGDATABASE value: "databricks_postgres" - name: PGPORT value: "5432" resources: - name: 'experiment' experiment: experiment_id: "<your-experiment-id>" permission: 'CAN_MANAGE' - name: 'postgres' postgres: branch: "projects/<your-project>/branches/<your-branch>" database: "projects/<your-project>/branches/<your-branch>/databases/<your-db>" permission: 'CAN_CONNECT_AND_CREATE'

Also update the targets section with your workspace host:

targets: dev: mode: development default: true workspace: host: https://<your-workspace>.cloud.databricks.com prod: mode: production workspace: host: https://<your-workspace>.cloud.databricks.com resources: apps: agent_openai_agents_sdk: name: agent-<your-app-name>
value_from vs value
value_from means the env var is automatically populated from the named resource binding (e.g., the postgres endpoint URL). value is a static string you set directly.

Step 8

Test Locally

Run the full application on your machine:

uv run start-app

This starts both the backend (port 8000) and frontend (port 3000). On first run, it automatically creates all database schemas and tables.

Open http://localhost:3000 and verify:

If all tests pass, your agent is ready to deploy to the cloud.
Step 9

Deploy to Databricks

# Validate configuration databricks bundle validate --profile DEFAULT # Deploy (uploads code, creates app) databricks bundle deploy --profile DEFAULT # Start the app databricks bundle run agent_openai_agents_sdk --profile DEFAULT
Deploy vs Run
bundle deploy uploads code and creates resources. bundle run starts the app. First startup takes 3-5 minutes.
Step 10

Grant Permissions to the App's Service Principal

Since you created the database tables locally (Step 8), the app's service principal needs access to them.

First, get the SP identity:

databricks apps get <your-app-name> --output json --profile DEFAULT | jq -r '.service_principal_client_id'

Then connect to Lakebase and run these grants:

-- Backend agent memory schema GRANT USAGE ON SCHEMA agent_openai_memory TO PUBLIC; GRANT ALL ON ALL TABLES IN SCHEMA agent_openai_memory TO PUBLIC; GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA agent_openai_memory TO PUBLIC; ALTER DEFAULT PRIVILEGES IN SCHEMA agent_openai_memory GRANT ALL ON TABLES TO PUBLIC; ALTER DEFAULT PRIVILEGES IN SCHEMA agent_openai_memory GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO PUBLIC; -- Frontend chat history schema GRANT USAGE ON SCHEMA ai_chatbot TO PUBLIC; GRANT ALL ON ALL TABLES IN SCHEMA ai_chatbot TO PUBLIC; GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA ai_chatbot TO PUBLIC; ALTER DEFAULT PRIVILEGES IN SCHEMA ai_chatbot GRANT ALL ON TABLES TO PUBLIC; ALTER DEFAULT PRIVILEGES IN SCHEMA ai_chatbot GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO PUBLIC; -- Drizzle migration journal GRANT USAGE ON SCHEMA drizzle TO PUBLIC; GRANT ALL ON ALL TABLES IN SCHEMA drizzle TO PUBLIC; GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA drizzle TO PUBLIC; ALTER DEFAULT PRIVILEGES IN SCHEMA drizzle GRANT ALL ON TABLES TO PUBLIC; ALTER DEFAULT PRIVILEGES IN SCHEMA drizzle GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO PUBLIC;
Why is this needed?
When you tested locally, tables were created under YOUR identity. The deployed app runs as a service principal — it needs explicit permission to access tables it didn't create. The ALTER DEFAULT PRIVILEGES ensures any future tables/sequences are also accessible.
Important
You must grant on SEQUENCES separately. Table grants alone are not enough. Without this, the service principal will get "permission denied for sequence" errors.
Alternative: Let the SP create everything
If you prefer no grants, drop all schemas before deploying and let the app's SP create them fresh (it owns what it creates). See the note in Step 8 about dropping schemas.
